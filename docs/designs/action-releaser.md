## Vision

### 10x Check

The 10x version is an org release platform: all versioning strategies, a `.release.yml` config per repo, plugin hooks for org-specific integrations (Slack, Jira, artifact signing), a release dashboard showing all releases across the org, and self-updating action versions. A single Go binary that teams trust completely.

**NOT in v1 scope from 10x vision:** Release dashboard, plugin hooks, artifact signing, self-updating action versions. These are deferred to future iterations.

## Key Strategic Decision: Go over Shell Scripts

The design doc's Approach A (composite action with shell scripts) was upgraded to Approach D (Go binary + composite action wrapper) during the CEO review. Rationale: the user wants org control and long-term extensibility. Go provides:
- Single compiled binary, zero runtime dependencies
- Strategy pattern via interfaces for versioning schemes
- Unit + integration testable
- Cross-compilation via GoReleaser
- No Docker pull overhead

## Key Technical Decisions (resolved during review)

- **GitHub API access:** `go-github` SDK (not `gh` CLI) — testable with mock server, no external runtime dependency
- **Date-rolling timezone:** UTC for reproducibility
- **git-cliff dependency:** External binary, exec'd via `os/exec`. Must be pre-installed on the runner. The composite action's first step installs git-cliff via `orhun/git-cliff-action@v4`.
- **Binary download:** `action.yml` downloads the pre-compiled Go binary from the action-releaser's own GitHub Releases, matching `runner.os` and `runner.arch`. Cached via `actions/cache` keyed on action version to avoid rate limits.

## Strategy Interface

```go
// VersionStrategy determines the next version based on existing tags and commits.
type VersionStrategy interface {
    // NextVersion returns the next version string (without tag prefix) given
    // the list of existing tags and a config. Returns ("", nil) if no release
    // is needed (semver: no conventional commits).
    NextVersion(tags []string, cfg Config) (string, error)

    // AlwaysReleases returns true if this strategy always produces a release
    // (date-rolling, numeric-rolling) vs. conditionally (semver).
    AlwaysReleases() bool
}
```

### Versioning Strategy Details

**Semver:**
- Uses `git-cliff --bumped-version` via `os/exec` to determine next version from conventional commits
- `git-cliff` emits a bare version string (e.g., `1.2.3` without prefix)
- **Bootstrap (no prior tag + has conventional commits):** Hard-code `0.1.0`
- **Bootstrap (no prior tag + no conventional commits):** Skip (no release)
- **No bump needed:** If `git-cliff --bumped-version` returns the same as the current tag, skip
- **Non-conventional commits only:** Same as no bump — skip release

**Date-rolling:**
- Format: `YYYY.MM.DD` (UTC). First release of the day has no suffix.
- Second release same day: `YYYY.MM.DD.2`, third: `YYYY.MM.DD.3`, etc.
- Always releases. If no new commits since last tag, changelog is empty.
- Bootstrap (no prior tag): First release is today's date.

**Numeric-rolling:**
- Tags are `{prefix}1`, `{prefix}2`, `{prefix}3`, etc.
- Increment latest numeric tag by 1. Bootstrap starts at `1`.
- Always releases. If no new commits since last tag, changelog is empty.

### Dry-Run Behavior

When `dry-run: true`:
- Calculates next version (runs full strategy logic)
- Generates changelog via git-cliff
- Writes `version`, `changelog`, `previous-version` outputs
- Sets `skipped` output correctly
- Does NOT create a git tag
- Does NOT push anything
- Does NOT create a GitHub release
- Sets `dry-run: true` output for downstream conditional steps

## `.release.yml` Config Schema

```yaml
# .release.yml — checked into repo root
version-strategy: semver    # semver | date-rolling | numeric-rolling
tag-prefix: "v"             # prefix for git tags
cliff-config: ""            # path to custom cliff.toml (optional)
draft: false                # create release as draft
prerelease: false           # mark as prerelease
```

**Priority chain:** Action inputs > `.release.yml` > built-in defaults. Action inputs always win when explicitly set.

## Action Inputs & Outputs

### Inputs

| Input | Description | Default |
|-------|-------------|---------|
| `version-strategy` | `semver`, `date-rolling`, `numeric-rolling` | `semver` |
| `cliff-config` | Path to custom cliff.toml | `""` (auto-detect) |
| `tag-prefix` | Prefix for git tags | `v` |

| `prerelease` | Mark as prerelease | `false` |
| `dry-run` | Calculate version + changelog but don't release | `false` |
| `github-token` | Token for releases. Needs `contents: write`. | `${{ github.token }}` |

### Outputs

| Output | Description |
|--------|-------------|
| `version` | Calculated version string (empty if skipped) |
| `changelog` | Generated changelog text (empty if skipped) |
| `tag` | Created git tag (empty if skipped or dry-run) |
| `release-url` | GitHub release URL (empty if skipped or dry-run) |
| `previous-version` | Previous tag. Empty on first release (changelog covers all commits). |
| `skipped` | `true` if no release needed, `false` otherwise |
| `dry-run` | `true` if dry-run mode, `false` otherwise |

### Shallow-Clone Guard

`gitutil.go` checks `git rev-parse --is-shallow-repository`. If `true`, the binary exits with code 1 and prints:
```
Error: shallow clone detected. Add 'fetch-depth: 0' to your actions/checkout step.
```
This is a fatal error — the binary does not attempt to fetch more history.

### Binary Download in action.yml

```yaml
# action.yml (simplified)
steps:
  - name: Install git-cliff
    uses: orhun/git-cliff-action@v4
    with:
      args: --version
    env:
      GITHUB_TOKEN: ${{ inputs.github-token }}

  - name: Download releaser binary
    shell: bash
    run: |
      VERSION="${{ github.action_ref }}"  # e.g., v1.0.0
      OS=$(echo "${{ runner.os }}" | tr '[:upper:]' '[:lower:]')
      ARCH="${{ runner.arch }}"
      [ "$ARCH" = "X64" ] && ARCH="amd64"
      [ "$ARCH" = "ARM64" ] && ARCH="arm64"
      CACHE_KEY="releaser-${VERSION}-${OS}-${ARCH}"
      # Check cache first
      if [ ! -f /tmp/releaser ]; then
        gh release download "$VERSION" \
          --repo tamedia-tech/action-releaser \
          --pattern "releaser_${OS}_${ARCH}" \
          --output /tmp/releaser
        chmod +x /tmp/releaser
      fi
    env:
      GITHUB_TOKEN: ${{ inputs.github-token }}

  - name: Run releaser
    shell: bash
    run: /tmp/releaser
    env:
      INPUT_VERSION_STRATEGY: ${{ inputs.version-strategy }}
      INPUT_TAG_PREFIX: ${{ inputs.tag-prefix }}
      INPUT_DRY_RUN: ${{ inputs.dry-run }}
      # ... etc
```

The calling workflow requires `permissions: contents: write`.

## Eng Review Amendments

**Architecture pivot (from eng review):** Changed from composite action + binary download to **Docker action**. All tools (releaser binary, git-cliff, git) baked into a Docker image on GHCR. Removes `orhun/git-cliff-action` dependency, `curl`/`gh` download steps, and caching logic. Simpler and self-contained.

**Monorepo partial failure:** If one package fails during a monorepo release, the action continues releasing other packages and reports partial failure with a non-zero exit code.

**Interface update:** `Config` struct gets a `Package` field for monorepo support. Strategies inspect `cfg.Package` for tag-pattern filtering.
